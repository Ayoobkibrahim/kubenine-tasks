1. Did the broken Dockerfile fail at build time, runtime, or both?
It likely failed at runtime. The Dockerfile syntax itself was valid, so Docker successfully created the image, 
but the application crashed or misbehaved when you actually tried to start the container.


2. What did you observe during the first docker build attempt?
The build completed successfully. Docker read the instructions, copied the file, installed Flask, 
and created the image without throwing any red error text.


3. What did you observe when running the broken container?
When trying to run it in detached mode (-d), it either crashed and immediately stopped (visible via docker ps -a), 
or the application started but was completely unreachable when trying to access localhost:5000 in the browser.


4. What did docker logs reveal?
It showed the exact application-level error. This usually reveals a Python crash (like a missing dependency because it 
wasn't installed properly) or shows that the web server is listening on the wrong internal network configuration


5. What did docker inspect reveal?
It acted like an X-ray. It revealed that the container was either in an "exited" state (crashed) or that the network 
port bindings were missing or configured incorrectly, preventing outside traffic from reaching the app.


6. What exactly was wrong in the original Dockerfile?
It manually ran RUN pip install flask instead of using the requirements.txt file. This means it just grabbed the newest, 
random version of Flask, ignored any specific version requirements, and completely missed any other potential dependencies your app needed.


7. Why is installing dependencies from requirements.txt better than installing them ad hoc?
It guarantees consistency. Ad hoc installs (like pip install flask) grab whatever the latest version is today, which might break your code tomorrow. 
A requirements.txt file locks in the exact versions, ensuring the app runs identically on every machine, every time.


8. Why must required files be copied into the image explicitly?
Because a Docker container is a completely isolated, empty box. It does not magically have access to the files on your laptop's hard drive. 
If you don't explicitly use the COPY command to put a file inside the image, it simply won't exist there.


9. What specific changes fixed the issue?
I had to update the COPY command to include requirements.txt (or just copy the whole directory) and replace the manual 
RUN pip install flask with RUN pip install -r requirements.txt to ensure the exact, correct environment was built.


10. What is the difference between a build-time problem and a run-time problem?

* Build-time problems happen when Docker is trying to construct your blueprint (the image). 
It usually means a typo in your Dockerfile or a file you are trying to COPY doesn't exist.

* Run-time problems happen after the image is built, when you actually start the container. The box was built fine, 
but the application inside the box crashed (e.g., bad Python code, missing database connection).


11. Why is docker logs important in container debugging?
It is your first line of defense. Because containers run in the background, you can't see the screen. docker logs lets you read 
the actual crash reports, print statements, and error traces generated by your application.


12. Why is docker inspect important in container debugging?
It shows you the hidden infrastructure. If the logs say the app is running perfectly, but you still can't reach it, docker inspect proves whether your port mappings, network IP addresses, and environment variables are actually set up correctly.


13. Why is docker exec useful during debugging?
It lets you break into the box. It opens a live terminal shell directly inside the running container so you can look around the file system, 
check if files were copied to the right folders, and manually test commands just like you would on a normal server.


14. Why should you apply minimal fixes instead of rewriting everything blindly?
Because blindly copying and pasting a "working" file doesn't teach you why the original one broke. Making minimal, 
precise fixes forces you to understand the exact root cause of the bug, which makes you a much sharper engineer.


15. How does this task help in real production Docker troubleshooting?
In the real world, production containers will fail. This task teaches you the exact standard operating procedure—check the state (ps), read the output (logs), check the config (inspect), 
and test inside (exec)—that senior DevOps engineers use to diagnose and fix live outages.
