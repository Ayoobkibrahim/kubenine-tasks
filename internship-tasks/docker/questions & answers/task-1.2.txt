# Task 1.2 - Run, Inspect, and Understand a Service Container

## Mandatory Questions

1. Why did the NGINX container keep running while the busybox container exited?
Containers only stay alive as long as their main background process is running. NGINX starts a continuous web server, so the container stays active. 
Busybox, by default, just runs a quick shell command and finishes immediately; once its job is done, the container exits.


2. What does detached mode mean in Docker?
Running a container in detached mode (using the -d flag) means it runs silently in the background. 
It frees up your current terminal window so you can keep typing other commands while the container does its work behind the scenes


3. What does -p 8080:80 mean exactly?
It is a port mapping rule. It tells Docker: "Take any traffic hitting port 8080 on my actual computer (the host) and forward it to port 80 inside the container."


4. What is the difference between host port and container port?
The Container port (80) is the isolated port inside the Docker network where the application (NGINX) is actively listening. 
The Host port (8080) is the exposed port on the physical machine (e.g., your laptop or an AWS EC2 instance). 


5. How did traffic from the browser reach the container?
When you type localhost:8080 in your browser, the request hits your host machine. Docker acts like a traffic cop; 
it intercepts that request on host port 8080 and directly routes it through a virtual bridge network to port 80 inside the container.


6. What is the purpose of docker logs?
It lets you see the text output and errors generated by the application running inside the container. 
It is always your first stop for troubleshooting if a container crashes or behaves unexpectedly.


7. What is the purpose of docker inspect?
It acts like an X-ray for your container. It outputs a highly detailed JSON file showing everything about the container's underlying 
configuration—including its internal IP address, mounted data volumes, port bindings, and environment variables.


8. What is the purpose of docker top?
It's like a mini Task Manager or Activity Monitor specifically for that container. 
It shows you the exact Linux processes currently running inside the container, along with their Process IDs (PIDs)

9. What is the purpose of docker exec?
`docker exec` allows you to run a new, secondary command inside an already running container. By using `-it` with a shell (like `sh` or `bash`), 
it effectively gives you an interactive terminal backdoor into the live container to inspect files, test networking, or debug issues without stopping the service.


10. What is the purpose of docker stats?
It provides a live, real-time dashboard of resource usage. It continuously streams exactly how much CPU, memory, 
and network bandwidth each running container is consuming at that exact moment.


11. What did you observe about CPU and memory usage for the container?
The NGINX container utilizes an incredibly small amount of resources—typically sitting near 0% CPU and using only a few megabytes of RAM. 
This highlights how lightweight containers are compared to full Virtual Machines.


12. Why is checking resource usage useful in production?
It prevents system crashes. By monitoring stats, you can catch memory leaks, identify CPU bottlenecks, and plan your infrastructure limits. 
If one container starts hogging resources, it can starve other critical applications running on the same server.


13. What happened when you ran docker stop?
Docker sent a graceful shutdown signal (SIGTERM "Signal Terminate") to the main process inside the container. The application finished whatever it was currently doing and safely shut down, 
moving the container's status from "running" to "exited."


14. Why does removing the container not remove the image?
Because a Docker image is a read-only template, while a container is just a temporary, running instance created from that template.
When you launch a container, Docker doesn't alter the original image. Instead, it takes the read-only image and adds a thin 'read-write' layer on top of it. 
Any data generated or changes made while the application is running happen strictly in that top layer.
When you run docker rm to remove the container, you are only deleting that temporary read-write layer. 
The underlying image remains completely untouched and safely stored on your system.


15. How does this task help later in Docker debugging and operations?
This task builds the exact muscle memory needed for real-world incident response. When a production application goes down or behaves strangely, 
these specific commands are the primary diagnostic toolkit used to find and fix the root cause.
