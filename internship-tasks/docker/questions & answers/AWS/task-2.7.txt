1. Why can the public EC2 instance be reached from the internet but the private one cannot?
A public instance is reachable because it has a public IP address and its subnet is wired directly to the Internet Gateway. 
A private instance cannot be reached because it lacks a public IP address, and its subnet is intentionally disconnected from the Internet Gateway, 
keeping it securely hidden from the outside world.


2. What is User Data and when exactly does it execute during the instance lifecycle?
User Data is a feature that allows to pass custom scripts or commands to an instance at launch to automate its initial configuration (a process known as "bootstrapping"). 
This is typically used to automatically install software, apply updates, download files, or start services without requiring manual human intervention.
It executes exactly once—during the final stage of the instance's initial boot process, right after the operating system has started for the very first time. 
It does not run again if stop, start, or reboot the instance later.


3. What happens if the User Data script has a bug — does the instance still launch?
Yes, the instance will still launch successfully and show as "Running." AWS doesn't stop the server if script fails; the instance will just boot up normally, 
but whatever software were trying to install (like Nginx) will simply fail or remain incomplete.


4. What problem does an Elastic IP solve that an auto-assigned public IP does not?
An auto-assigned public IP changes every time stop and start a server. An Elastic IP solves this by giving a permanent, 
static IP address that stays attached to the instance no matter how many times stop or reboot it. 
This ensures application remains consistently reachable.

5. How does the private instance reach the internet for software updates if it has no public IP?
It uses a NAT (Network Address Translation) Gateway located in the public subnet. The NAT Gateway acts like a proxy or a middleman: it takes the private instance's request, 
goes out to the internet to get the update, and hands it back to the private instance, all without exposing the private server to the outside world.


6. Why is the private security group configured to allow SSH only from the public security group, not fromr IP?
Because a private instance has no public IP, computer cannot reach it directly over the internet. Configuring it to only accept SSH from the public instance creates a secure "Bastion Host" architecture, 
forcing to securely log into the public server first before can "jump" into the hidden private server.


7. What would happen if you accidentally deleted the NAT Gateway while the private instance was running?
The private instance would stay running, but it would instantly lose all access to the outside internet. It would no longer be able to download updates, communicate with external APIs, 
or ping external websites until a new NAT Gateway was created.


8. If HTTP port 80 is not open in the security group, what happens when you try to access Nginx?
Browser will just load forever and eventually say "Connection Timed Out." Because Security Groups act as a firewall that blocks everything by default, 
AWS will silently drop web traffic before it ever even reaches the Nginx server.
