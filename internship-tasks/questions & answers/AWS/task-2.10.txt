what is Versioning
S3 Bucket Versioning is a data protection feature in Amazon S3 that allows you to keep multiple variants of an object in the same bucket. 
When enabled, it preserves, retrieves, and restores every version of every object stored in your bucket.

1. What does versioning protect against that "Block Public Access" does not?
"Block Public Access" is a security perimeter that protects against external threats (preventing the outside internet from seeing my data). 
Versioning is an operational safeguard that protects against internal human error or application bugs (like accidentally deleting a database backup or overwriting a file with bad code).


2. What is a delete marker and why doesn't a delete actually remove the object?
A delete marker is essentially a 'soft delete.' It hides the file from the main view without actually destroying the data, 
which allows to instantly recover the file if realize it was deleted by mistake.


3. Can you disable versioning once it is enabled? What happens instead?
No, once versioning is enabled, it cannot be permanently disabled. It can only be 'Suspended.' If suspend versioning, S3 will simply stop keeping new versions of files moving forward. 
However, all the historical versions that were already created will remain in the bucket indefinitely until manually delete them or a lifecycle rule clears them out.


4. What problem do lifecycle rules solve in production?
S3 Lifecycle rules automate the management of objects over time. In production, accumulate massive amounts of data—like old logs or historical file versions. Instead of paying an engineer to manually clean out buckets,
They are configured policies that automatically transition objects to cheaper storage classes (like from S3 Standard to S3 Glacier) as they age, or permanently expire (delete) objects and historical versions when they are no longer needed.


5. What is encryption at rest and why is it important even inside AWS?
Encryption at rest is the technique of protecting data by encoding it while it is stored on physical hard drives, databases, or backups—making it unreadable to unauthorized users without the correct decryption key.
Why Encryption at Rest is Important Inside AWS? 
Even though AWS operates within a highly secure, controlled environment, encryption at rest is crucial for a "defense-in-depth" security strategy, where multiple layers of security prevent data breaches.
it adds that crucial final layer of protection, ensuring sensitive data remains entirely unreadable to anyone who doesn't hold the keys, while satisfying mandatory industry regulations.


6. What is the difference between SSE-S3 and SSE-KMS? When would you choose KMS?
SSE-S3 (S3-Managed Keys): This is the simple, default, and free option. AWS handles everything invisibly in the background. They generate, securely store, and rotate the encryption keys.
SSE-KMS (Key Management Service): This is the customer-controlled option. manage the encryption keys myself through AWS KMS. It costs a small fee, but it allows you to set specific access policies on the key itself and creates an audit log every single time the key is used.
I would choose KMS for highly sensitive production data because it provides a complete audit log of decryption activity and enforces strict separation of duties.


7. If versioning is enabled and you have 100 versions of a 1GB file, how much storage are you paying for?
Paying for exactly 100GB of storage for all versions, The reason is AWS S3 calculates storage bill based on the total aggregate size of all objects inside the bucket, 
which explicitly includes every single historical version of a file.
