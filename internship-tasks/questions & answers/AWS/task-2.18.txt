1. What is the main difference between Layer 4 and Layer 7 load balancing?
Layer 4 load balancing (NLB) routes traffic purely based on network data, like IP addresses and TCP/UDP ports, without looking at the payload. 
Layer 7 (ALB) is "application-aware," it actively inspects the HTTP/HTTPS requests—including headers, cookies, and URL paths—to make intelligent routing decisions.



2. Why can't NLB perform path-based routing?
An NLB operates strictly at Layer 4 of the network stack, It only understands raw connection data like IP addresses and port numbers. Path-based routing requires inspecting the actual HTTP request, 
in the URL which happens up at Layer 7. Because an NLB doesn't open the packets to read that HTTP application data, it simply has no way of knowing which path was requested, making path-based routing impossible.


3. When would you use NLB instead of ALB? Give a specific example.
I would choose an NLB when dealing with extreme performance requirements, non-HTTP traffic, or when we absolutely need a static IP address for firewall whitelisting.
For example, if I'm setting up a backend for thousands of IoT smart devices that stream raw TCP telemetry data continuously, an ALB wouldn't work because it expects standard HTTP web traffic. 
I would use an NLB to efficiently process those raw network connections at scale without adding any application-level delay.


4. Why is NLB considered higher performance than ALB?
An NLB acts much more like a direct network router. It doesn't spend compute cycles opening, inspecting, or modifying HTTP headers and payloads; it simply takes the raw TCP/UDP connection and forwards it at the network level, 
resulting in ultra-low latency and the ability to handle millions of requests per second.


5. Can NLB terminate SSL/TLS like ALB? What's different?
Yes, A Network Load Balancer can perform SSL/TLS termination, which means it handles the heavy computing work of decrypting secure traffic so backend servers don't have to.
When an ALB decrypts traffic, it looks inside the actual HTTP request like the URL path or headers to make smart routing decisions. When an NLB decrypts traffic, it simply takes that raw, 
decrypted TCP data stream and forwards it straight to servers at blinding speed, without ever inspecting the application-level data.


6. Why is static IP support important in some architectures?
In many enterprise environments, security rules are extremely strict. External partners or legacy corporate firewalls often operate using 'whitelists'—meaning they only allow network traffic from specific, pre-approved IP addresses.
Because an Application Load Balancer constantly changes its IP addresses behind the scenes to manage scale, it breaks these rigid security rules. A Network Load Balancer, however, allows to attach a permanent, static IP address (Elastic IP). 
This ensures application's public address never changes and can be reliably whitelisted by those external systems without causing connection drops.


7. You have a gRPC microservice that needs low latency. Which load balancer would you use and why?
For a gRPC microservice, I would use an Application Load Balancer (ALB) because it natively supports HTTP/2 and gRPC routing. While an NLB is technically faster at the network level, gRPC uses long-lived connections that an NLB cannot 'see' inside; 
an ALB can inspect the individual requests within those connections and balance them across all backend servers to prevent hotspots. This ensures that even with a single persistent connection, traffic is distributed evenly and remains highly available.
