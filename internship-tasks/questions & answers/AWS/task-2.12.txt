1. Why should the root user never be used for daily operations?
The root user has absolute, unrestricted access to every part of it AWS account, including billing, changing support plans, and closing the account entirely. 
If root credentials are compromised or a mistake is made, the entire business is at risk. should only use the root account to create it first administrative IAM user, 
enable MFA, and then lock the root account away.


2. What is the difference between an IAM user and an IAM role?
* An IAM User is a specific identity for a single person or application. It has permanent, long-term credentials (like a password to log into the console, or static access keys for code).
* An IAM Role is an identity with temporary, short-term credentials. It is not tied to a specific person. Instead, it is "assumed" by a user, an EC2 instance, or another AWS service just for the time they need it to do a job.


3. What is the difference between an AWS managed policy and a customer-managed policy?
An AWS Managed Policy is a pre-built template created and maintained by Amazon (like AmazonS3ReadOnlyAccess). cannot edit them, but they are quick and easy to attach. 
A Customer-Managed Policy is a custom JSON policy write itself. It gives precise, granular control over exactly which actions are allowed on it specific resources.


4. What happens when one policy allows an action and another policy explicitly denies it?
The explicit deny always overrides the allow and blocks the action. AWS IAM evaluation logic defaults to an implicit deny. If a policy grants an "Allow," access is granted. However, 
if any policy attached to the user contains an "Explicit Deny" for that same action, it instantly overrides all allows. One deny trumps a hundred allows.


5. Why is AdministratorAccess dangerous in production?
AdministratorAccess grants absolute, unrestricted permission to do anything to any resource in it AWS account. In a production environment, this is incredibly dangerous because it maximizes the "blast radius" of any mistake or security breach. 
If a developer's credentials are stolen, or if they accidentally run the wrong script, there are no guardrails to stop them from permanently deleting live databases or wiping out core infrastructure.


6. What is least privilege and how do you apply it in practice?
The Principle of Least Privilege means granting a user or service only the minimum level of access required to perform its specific job, and absolutely nothing more. In practice, apply this by avoiding broad AWS managed policies 
and instead writing custom policies that restrict actions to exact resource ARNs (e.g., allowing s3:GetObject  instead of all buckets).

what is ARNs
An Amazon Resource Name (ARN) is a unique identifier for AWS resources, acting as a standardized naming convention to unambiguously identify specific assets like EC2 instances, S3 buckets, or IAM roles across services, regions, and accounts. 
ARNs are essential for creating IAM policies, API calls, and automation scripts

7. Why is MFA critical for the root account? What risk does it mitigate?
Because the root account has absolute power over it entire AWS environment, relying on just a password creates a massive single point of failure. Multi-Factor Authentication (MFA) requires a second, time-sensitive code generated by a physical device (like it phone). 
It completely mitigates the risk of stolen credentials—even if a hacker steals, guesses, or phishes it root password, they cannot log in without physically holding it MFA device.


8. If an IAM user's password is compromised but MFA is enabled, can the attacker access the console?
No. To access the AWS Management Console, authentication requires both the first factor (the password) and the second factor (the MFA token). Even if the attacker knows the password perfectly, the login process will halt and deny access without the six-digit code from the physical MFA device.
