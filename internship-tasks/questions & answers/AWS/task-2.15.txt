1. What is AWS STS and what does it do?
AWS STS (Security Token Service) is the web service responsible for creating and providing trusted users with temporary security credentials. Instead of using permanent passwords or access keys, STS acts like a ticket booth, 
handing out short-lived, dynamically generated "passes" that grant temporary access to AWS resources.


2. What does the AssumeRole action return?
When successfully call sts:AssumeRole, STS returns a package of temporary credentials consisting of three exact things: an Access Key ID, a Secret Access Key, and a Session Token, 
along with an exact expiration time for when those credentials will stop working.


3. Why are temporary credentials safer than long-lived access keys?
Temporary credentials are self-destructing. If a hacker manages to steal them, they only have a very short window (usually 15 minutes to an hour) to use them before they become completely invalid. 
Long-lived access keys never expire automatically, meaning a stolen key gives an attacker permanent access until manually realize the breach and delete the key.


4. What is the difference between the trust policy and the permission policy on a role?
The Trust Policy controls who or what is allowed to assume the role (for example, trusting the EC2 service or a specific user). 
The Permission Policy controls what actions the role is allowed to perform once it is assumed (for example, reading from a specific S3 bucket). 
Both must be configured correctly for access to work.


5. What happens when temporary credentials expire?
Access is instantly and automatically denied. Any API calls or CLI commands using those expired credentials will fail with an "Access Denied" or "ExpiredToken" error. 
To regain access, the user or application must make a brand new AssumeRole request to STS to get a fresh set of credentials.


6. If the trust policy allows a user but the user doesn't have sts:AssumeRole permission, can they assume the role? Why?
No, they cannot. IAM role assumption requires a two-way handshake. The role must explicitly state "I trust this user to assume me" (Trust Policy), 
AND the user must explicitly have an IAM policy attached to them saying "I am allowed to call sts:AssumeRole on this specific role." If either side is missing, the action is blocked.


7. Why is the session token mandatory when using temporary credentials?
The Session Token is the cryptographic proof that these credentials were dynamically generated by STS and are tied to a specific expiration timer. 
If try to use the temporary Access Key and Secret Key without the Session Token, AWS will assume are trying to use permanent IAM user keys, 
fail to find them in the database, and reject request.


8. How does STS relate to what happens behind the scenes when EC2 assumes a role (Task 2.13)?
In Task 2-13, I didn't have to manually call STS because the EC2 Instance Metadata Service (IMDS) did it Behind the scenes, 
the EC2 service continuously calls sts:AssumeRole on behalf, retrieves the temporary credentials (Access Key, Secret Key, Session Token), 
and injects them securely into the metadata server (169.254.169.254) so application can use them transparently.
