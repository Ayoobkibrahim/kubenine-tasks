1. Why should secrets never be stored in application code or configuration files?
Storing secrets in code or config files is a massive security vulnerability. If committed to version control (like GitHub), anyone with access to the repository, 
build logs, or container image layers can easily extract these hardcoded credentials and compromise entire system.


2. How does Secrets Manager encrypt secrets at rest?
AWS Secrets Manager encrypts secrets at rest using AWS KMS (Key Management Service). It uses a KMS key to mathematically scramble the key-value pairs so the 
data remains unreadable on AWS's physical disks without the proper decryption key and IAM permissions.


3. What IAM action is required to retrieve a secret value?
To fetch the actual decrypted value of a secret via the API or CLI, the specific IAM action required is secretsmanager:GetSecretValue.Without this permission, 
an application or user cannot read the secret from Secrets Manager.


4. Why should IAM policies avoid "Resource": "*" for secrets access?
Using * violates the Principle of Least Privilege because it grants the application access to every secret in AWS account. must explicitly scope the 
policy to the exact ARN of the required secret, ensuring a compromised application cannot steal credentials belonging to other databases or services.



5. What is the difference between Secrets Manager and SSM Parameter Store?
While both store configuration data, Secrets Manager is purpose-built for sensitive credentials and features built-in, automatic secret rotation like changing a database password every 30 days. 
SSM Parameter Store is primarily for plain-text configuration (like endpoint URLs), and while it can store secure strings, it lacks native, automatic rotation capabilities.


6. How would a containerized application retrieve secrets at runtime without hardcoding them?
Instead of hardcoding passwords in the image, the container assumes an IAM Role and makes an API call to AWS Secrets Manager at startup. This securely pulls the decrypted credentials 
directly into the application's memory (RAM), ensuring the password is never saved to a file or exposed.


7. What happens if an EC2 instance's IAM role does not have permission to read a specific secret?
The API call will be explicitly denied by AWS. The application running on the EC2 instance will receive an AccessDeniedException 
from Secrets Manager and will consequently fail to start or authenticate with its backend services.
