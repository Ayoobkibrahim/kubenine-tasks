1. What is the difference between a chart you write from scratch and one generated by helm create?
A chart written from scratch contains only the files and templates we manually create based on our requirements, while a chart generated using helm create comes with 
default boilerplate templates and sample configurations automatically. Writing a chart from scratch helps us understand Helm structure, templating, and chart design more deeply.

2. How does Go template syntax work in Helm templates?
Helm uses Go template syntax inside {{ }} brackets to dynamically insert values into Kubernetes YAML files. During rendering, 
Helm replaces these template expressions with actual values from values.yaml or built-in Helm objects.

3. Why should nothing be hardcoded in chart templates?
Nothing should be hardcoded because hardcoding reduces flexibility and makes charts difficult to reuse across environments 
like development, staging, and production. Using configurable values from values.yaml makes the chart reusable, maintainable, and easy to manage.

4. What is the purpose of _helpers.tpl?
_helpers.tpl is used to store reusable helper templates and functions that can be shared across multiple template files. 
It helps reduce duplication and keeps resource naming, labels, and common logic consistent throughout the chart.

5. How does a ConfigMap created by a Helm template differ from a plain ConfigMap manifest?
A plain ConfigMap manifest contains fixed static values directly written in YAML, while a Helm-templated ConfigMap dynamically generates values using values.yaml. 
This allows configuration to change based on different environments without modifying the template itself.

6. What does {{ .Values.image.repository }} do?
{{ .Values.image.repository }} reads the image.repository value from values.yaml and inserts it into the template during rendering. 
This makes the container image configurable instead of hardcoding it inside the Deployment template.

7. How does values-prod.yaml override defaults without modifying values.yaml?
Helm automatically merges values-prod.yaml with the default values.yaml during install or upgrade operations. 
Any matching values in values-prod.yaml temporarily override the defaults without changing the original values.yaml file.

8. What is the difference between helm install and helm upgrade --install?
helm install only creates a new release and fails if the release already exists, while helm upgrade --install upgrades an 
existing release or installs it automatically if it does not exist. This makes helm upgrade --install very useful in CI/CD pipelines.

9. How would you pass values at install time without a values file?
Values can be passed directly during installation using the --set flag in the Helm command. This allows quick configuration 
changes without creating a separate values file.

10. Why is a well-designed values.yaml important for chart reusability?
A well-designed values.yaml centralizes all configurable settings in one place and makes the chart easy to customize for multiple environments. 
It improves reusability, maintainability, and consistency while reducing the need to modify templates directly.
