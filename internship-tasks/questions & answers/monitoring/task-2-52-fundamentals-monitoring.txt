1. What is the difference between a Counter and a Gauge? Why does it matter which type a metric is?
A Counter metric only increases over time and resets when the application restarts, while a Gauge metric can increase or decrease freely depending 
on the current state of the system. This distinction is important because Counter metrics should be queried using functions like rate() or increase() 
to calculate changes over time, whereas Gauge metrics can be used directly to show current values such as memory usage or temperature.

2. Why does Prometheus use a pull model? What is the main advantage over a push model?
Prometheus uses a pull model where it periodically scrapes metrics from targets instead of targets sending metrics themselves. 
The main advantage of this approach is that Prometheus can automatically detect whether a target is healthy or unreachable and 
centrally control scraping intervals, making monitoring more reliable and easier to manage.

3. What is a scrape target? How does Prometheus know which targets to scrape?
A scrape target is any application, service, or Kubernetes component that exposes metrics through an HTTP /metrics endpoint. 
Prometheus discovers these targets using methods such as Kubernetes service discovery, ServiceMonitors, PodMonitors, 
or static scrape configurations defined inside the Prometheus configuration.

4. What happens to metric values in Prometheus when a target goes down?
When a target goes down, Prometheus stops receiving fresh metric samples from that target because scraping fails. 
Existing metric data remains temporarily in the time series database but eventually becomes stale and disappears 
from recent query results if the target remains unavailable.

5. Why should you never use a raw Counter value in a graph — why must you always use rate() or increase()?
Raw Counter values continuously increase over time and are not useful for understanding actual activity because they only show cumulative totals. 
Functions like rate() and increase() convert Counter data into meaningful values such as requests per second or total requests during a specific time range, 
which are much easier to analyze in monitoring dashboards.

6. What is a label in Prometheus? Give an example of how labels let you filter a query to a specific pod.
Labels are key-value pairs attached to metrics that provide additional context such as pod name, namespace, HTTP method, or instance information. 
For example, a query like http_requests_total{pod="nginx-1"} filters the metric results so that only requests from the nginx-1 pod are returned.

7. What does sum() by (namespace) do? What would happen if you removed the by (namespace) clause?
sum() by (namespace) groups and aggregates metric values separately for each Kubernetes namespace, allowing you to view individual totals per namespace. 
If the by(namespace) clause is removed, Prometheus combines all matching metrics together and returns one overall aggregated result for the entire cluster.

8. What is a histogram in Prometheus? What are the _bucket, _sum, and _count suffixes?
A histogram metric records observations such as request durations and stores them inside configurable value ranges called buckets. The _bucket suffix 
stores counts for each bucket range, _sum stores the total sum of all observations, and _count stores the total number of observations recorded.

9. What does histogram_quantile(0.99, ...) compute and when would you use it in production?
histogram_quantile(0.99, ...) calculates the 99th percentile latency, meaning 99% of requests completed faster than the returned value. 
In production environments, this is commonly used to identify slow API requests and measure user-facing performance during peak traffic or system load.

10. What is the up metric and how can it be used to detect a scrape target going down?
The up metric is automatically generated by Prometheus for every scrape target to indicate whether the target is reachable and successfully scraped. 
A value of 1 means the target is healthy, while a value of 0 means the target is down or scraping failed, making it useful for alerting and monitoring target availability.

11. What is label cardinality and why can high cardinality cause problems in Prometheus?
Label cardinality refers to the number of unique metric series created by different combinations of labels and label values. High cardinality increases memory usage, 
storage consumption, and query execution time, which can significantly slow down Prometheus and increase infrastructure costs.

12. How do you find out what metrics a specific target is exposing?
You can find all metrics exposed by a target by directly accessing its /metrics endpoint through a browser, curl command, or port-forwarded service URL. 
This endpoint displays every metric currently exposed by the application along with labels, metric types, and current values.
